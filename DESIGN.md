> 数学が苦手な中学生のための、つまづき解決に全振りした学習アプリ。
> math-dialogue（RPG型）とは別アプリ。問題生成・スキル・診断・はいち動画は「共有エンジン」として流用する。
> ステータス：構想確定 → 詳細設計（本書）。実装は未着手。

## 1. プレスリリース（完成像 ＝ Working Backwards）

**見出し**：「とけた！」— 数学ぎらいが、いつのまにか『自分で解けた』に変わる学習アプリ

**サブ**：わからない“まさにその一歩”をその場で解決。最後は必ず自分の手で解けるから、「できた！」が毎日たまる。

**本文**：
数学が苦手な人の多くは「わからない→止まる→放置→もっと嫌い」の悪循環にいる。「とけた！」は、つまづいた一歩を**お手本→穴うめ→自力**の順に足場を外しながら導き、**最後の一手は必ず本人**が出す。解いているうちに「自分がどこで止まりやすいか（符号ミス等）」が見え、その弱点だけが自動で多めに練習される。だから**このモードを解き続けるだけで、気づけばスキルが上がっている**。失敗に減点もタイマーもペナルティもない。あるのは小さな成功の連続だけ。

**顧客の声**：「わからない問題が“怖くなく”なった。ヒントで終わりじゃなくて最後は自分で解けるから『できた！』ってなる。先生にも褒められて、もう1問やりたくなる。」

## 2. FAQ

### 2-1. 顧客（生徒・保護者）向け
- **Q. 何ができる？** A. 1問ずつ、わからなければお手本→穴うめ→自力の順に助けが出て、必ず自分で解けるところまで導く。続けるだけで弱点が減る。
- **Q. math-dialogue と何が違う？** A. あちらはバトルや装備で“ガンガン進む”ゲーム。こちらは飾りを削ぎ、**つまづき解決と「わかった」だけに集中**したシンプルな作り。
- **Q. 間違えたら怒られる／減点される？** A. 一切なし。間違いは「つまづきの発見」として歓迎され、その場で直せる。
- **Q. ネットや費用は？** A. MVPは**完全オフライン・無料で動く**（問題はアプリ内生成）。動画やAIは後の段階で任意追加。

### 2-2. 社内（開発）向け
- **Q. なぜ別アプリ？** A. 数学ぎらいは“盛りだくさん”で離脱する。RPGの文脈と混ぜず、Less is More を貫くため。
- **Q. 一番のリスクは？** A. 「お手本ステップ」「誤答パターン付き4択」を全範囲で用意するコスト。→ MVPは正負の数に限定し、テンプレ自動生成で吸収。
- **Q. AI／サーバーは要る？** A. MVPは不要（完全静的）。AIと一緒に解く機能は将来の任意拡張（API＝コスト発生）。
- **Q. 著作権（はいち動画）は？** A. はいち動画はライセンス許諾済み資産（math-dialogue の `haichiCourse`）。Phase3で「つまづき箇所だけ」接続する想定。
- **Q. 成功の定義は？** A. §10 KPI。核は「自力到達率」と「再訪/継続」。

## 3. デザインプリンシプル

**北極星：つまづきを、その場で、自力で解けるに変える。**

迷ったら全部これに従う。具体5原則：
1. **1画面・1アクション**（認知負荷を最小化。ごちゃごちゃで離脱させない）
2. **失敗を罰しない**（タイマー・減点・×演出なし）
3. **毎回、必ず小さく勝たせる**（自力ゴールを保証する＝下記の不変条件）
4. **説明は「次の1歩」だけ**（解法全体を見せない。今つまった行だけ）
5. **ほめは具体的に**（「符号ミスが消えたね」＝成長を言語化）

## 4. コア体験ループ（適応はしご）

### 4-1. ループ
ちょうど良い1問 → やってみる → （正解＝ほめて少し上げる／つまづき＝助けのはしご）→ 自力で解けた → 記録（つまづきマップ）→ 復習 → 戻る

### 4-2. 助けのはしご（5段）
1. **ヒント**（やさしい一言）
2. **お手本**（全ステップを見せる）
3. **穴うめ**（一部の行を本人が埋める）
4. **ミニ動画**（つまづき箇所だけ。Phase3）
5. **やさしい類題**（もっとやさしい同型）

> MVPの中核は **2お手本 → 3穴うめ → 自力**。1ヒント・5類題は補助、4動画は後段。

### 4-3. 開始段は「習熟度」で自動決定（適応）
全員がお手本からではない。`skillStats[skill].m`（0〜1）で開始段を変える：
- `m < 0.35` → **お手本**から（足場フル）
- `0.35 ≤ m < 0.70` → **穴うめ**から
- `m ≥ 0.70` → **自力**から

### 4-4. 不変条件（必ず守るルール）
- **最後の一手は必ず本人が出す**（お手本でも、最終の答えだけは本人が選ぶ/入力）。
- **つまづいたら1段だけ降ろす**（自力→穴うめ→お手本）。一気に答えを見せない。
- **降りても、同型でもう一度“自力”に戻れるまで続ける**（＝必ず「自分で解けた」で終わる）。

## 5. 画面遷移（全画面）

画面は本質的に3つ＋演出。タブやメニューは作らない。

```
[ホーム/つづきから]
      │ ▶ とくよ
      ▼
[とくよ画面] ──つまづき──▶ [助け：お手本→穴うめ→自力]──自力で正解──▶ [ほめ演出]
      │  ▲                                                              │
      │  └───────────────── 次の1問（同型 or 次の型）◀────────────────┘
      ├─ ふりかえり ▶ [ふりかえり画面（復習）]
      └─ マップ     ▶ [つまづきマップ]
```

### 5-1. ホーム（つづきから）
- 目的：迷わせず即スタート。
- 要素：大きな「▶ とくよ（◯◯の続き）」、今日のひとこと（ほめ）、小さく「ふりかえり」「つまづきマップ」。
- 原則：選択肢は最小。学年/単元選びは奥に隠す（既定は前回の続き）。

### 5-2. とくよ画面（主役）
- 目的：1問に集中。
- 要素：問題1問（大きく）／4択 or 入力／下に控えめな「ヒントがほしい」。タイマー・スコア・×演出なし。
- 遷移：正解→ほめ→次の1問。つまづき（間違い or ヒント要求）→ 助け画面へ。

### 5-3. 助け画面（お手本／穴うめ／自力）
- 目的：足場を出し、最後は本人に解かせる。
- お手本：全ステップ＋最終の答えだけ本人が選ぶ。
- 穴うめ：1〜2行を空欄に。空欄だけ本人が埋める。
- 自力：問題だけ。解けたら同型をもう1問（足場ゼロで再確認）。
- 共通：誤答を選ぶと**その場で原因を一言**（§6 診断）。「もう1段やさしく」ボタンで降段。

### 5-4. ふりかえり画面（復習）
- 目的：解いた所を間隔をあけて再出題し定着。
- 要素：今日直したつまづき一覧（「符号ミス→直せた」を1行）。期限が来た問題を自動で再出題。

### 5-5. つまづきマップ
- 目的：弱点の見える化＋具体的なほめ。
- 要素：スキル別の状態（要練習/練習中/克服）。「先週より符号ミス−70%」等の差分メッセージ。

### 5-6. ほめ演出（共通）
- 目的：「できた！もっと！」を作る。
- 要素：短い祝福＋具体的な一言（速くなった/ミスが消えた）。派手すぎず、毎回出す。

## 6. データ仕様

### 6-1. 共有エンジン（math-dialogue から流用＝そのまま使う）
- `engine/generator`（同型問題を無限生成）
- `data/*` 問題バンク（正負・文字式…）
- `engine/skills`（スキルDAG：`S-NEG-xxx` 等）
- `engine/scoring`（`isCorrect`／`makeChoices`）
- `engine/weakness`・`engine/tagAnalysis`・`engine/insight`（つまづき診断）
- `data/haichiCourse`（つまづき箇所のミニ動画。Phase3）

> 切り出し方針：これらを `@toketa/engine`（共有パッケージ or 共有フォルダ）として独立させ、math-dialogue と「とけた！」の両方が依存する。UI 依存を持たない純ロジックのみを移す。

### 6-2. 新規に足す“2つ”だけ（ここが肝）

#### (A) お手本ステップ `workedSteps`
問題テンプレに「どう解くか」と「どの行を空欄にできるか」を持たせる。正負・一次式は **build から自動生成可**（手作業ほぼ不要）。

```js
// 問題テンプレ（既存の build を拡張）
const p = {
  id: "neg-add-1",
  skill: "S-NEG-007",
  build(rng) {
    const a = rng(1, 9), b = rng(1, 9);
    const ans = -(a + b);
    return {
      q: `−${a} +（−${b}）`,
      ans,
      // お手本ステップ（blank:true の行は穴うめで空欄にできる）
      steps: [
        { say: "同符号どうしのたし算", value: null },
        { say: `絶対値をたす：${a}+${b}`, value: a + b, blank: true },
        { say: "符号はそのまま", value: "−", blank: true },
        { say: "答え", value: ans, blank: true, final: true },
      ],
      // 誤答パターン付き4択（下記 B）
      distractors: makeMisconceptionChoices({ a, b, ans }),
    };
  },
};
```

#### (B) 誤答パターン付き4択 `distractors`（診断の心臓）
`makeChoices` を拡張し、**まちがいの選択肢に“つまづきの正体（tag）”を仕込む**。選んだ瞬間に原因を言い当てる。

```js
// 例：−a +（−b）
function makeMisconceptionChoices({ a, b, ans }) {
  return [
    { val: ans,        tag: null,        why: null },                       // 正解 −(a+b)
    { val: a + b,      tag: "sign-flip", why: "符号を「＋」にしちゃったかも" }, // +(a+b)
    { val: -(b - a),   tag: "subtract",  why: "ひき算と混ざったかも" },        // 引いてしまう
    { val: -Math.max(a,b), tag: "abs-only", why: "大きい方の絶対値だけ見たかも" },
  ];
}
```

診断タグはスキルに紐づく共通辞書で管理（再利用）：
```js
const MISCONCEPTIONS = {
  "sign-flip": { label: "符号の取り違え", coach: "（−）＋（−）は答えも「−」だよ" },
  "abs-only":  { label: "絶対値だけ計算", coach: "符号も忘れずに付けよう" },
  // …
};
```

### 6-3. 進捗の保存（プレイヤー状態・新アプリ側）
math-dialogue の `localStore` 方式（ローカル＋将来 supabase 差し替え）を踏襲。新規フィールド：

```js
{
  skillStats: { "S-NEG-007": { m: 0.42, n: 12, last: "2026-06-18" } }, // 共有：習熟度(Elo)

  cleared:    { /* 自力到達した型 */ },
  praiseSeen: { /* 出したほめの履歴（重複回避） */ },
}
```

## 7. 適応・診断ロジック（疑似コード）

```js
// 出題：弱点を多めに、ちょうど良い難易度で1問
function nextProblem(player) {
  const skill = pickSkillWeightedByWeakness(player.skillStats, player.stumbles); // weakness流用
  const tmpl  = pickTemplateForSkill(skill);
  return { skill, problem: tmpl.build(rng) };
}

// 開始段：習熟度で決定
function startRung(m) {
  if (m < 0.35) return "example"; // お手本
  if (m < 0.70) return "fill";    // 穴うめ
  return "solo";                  // 自力
}

// 回答時
function onAnswer(choice, ctx) {
  if (choice.tag) {                          // 間違い＝つまづき
    record(ctx.player.stumbles, ctx.skill, choice.tag);
    showCoach(MISCONCEPTIONS[choice.tag].coach);  // その場で原因を一言
    ctx.rung = downRung(ctx.rung);           // 1段だけ降ろす（solo→fill→example）
    return "retry-same-type";                // 同型でやり直し（必ず自力ゴールまで）
  }
  // 正解
  ctx.player.skillStats[ctx.skill] = elo(ctx.player.skillStats[ctx.skill], +1);
  if (ctx.rung === "solo") scheduleReview(ctx.player, ctx.problem);  // 自力正解だけ復習へ
  praise(ctx);                               // 具体的なほめ
  return "next";
}

// 復習（間隔反復・簡易）：1日→3日→7日…、つまづけば間隔リセット
function scheduleReview(player, problem) {
  push(player.reviewQueue, { problemId: problem.id, skill: problem.skill, dueDate: today()+1 });
}
```

## 8. ほめ・モチベ設計（数学ぎらい向け）
- **比較は過去の自分とだけ**（他人・偏差値を出さない）。
- ほめは**具体的・行動ベース**：「速くなった」「符号ミスが消えた」「3日続いた」。
- ご褒美は最小限：派手なガチャやバトルは入れない（math-dialogue 側の役割）。
- 連続日数は“プレッシャー”にしない（途切れても優しく再開）。

## 9. MVP スコープと段階リリース
1. **MVP**：正負の数（加減）1テーマで、**適応はしご（お手本→穴うめ→自力）＋誤答診断＋つまづきマップ＋ほめ**。完全オフライン・無料。
2. **+範囲**：正負（乗除・四則）→ 文字式 → 一次方程式（テンプレ自動生成で拡張）。
3. **+動画/復習強化**：はしご④（はいちのつまづき箇所動画）＋間隔反復の本格化。
4. **+一緒に**：はしご③（AIと1ステップずつ／サーバー任意）。

## 10. 成功指標（KPI）
- **自力到達率**：助けに入った問題のうち、最終的に「自力」で正解できた割合（目標 ≥ 95%）。← 北極星の直接指標
- **つまづき解消**：同一タグの誤答が、再出題で減っているか（週次−◯%）。
- **継続**：翌日再訪率／1週間継続率。
- **主観**：「わかった」「もう1問やりたい」に当たる簡易フィードバック（任意の😊タップ等）。

## 11. リスクと対策
| リスク | 対策 |
|---|---|
| お手本/誤答データの作成コスト | 正負・一次式は build から自動生成。手作りは難所だけ。 |
| 助けに頼りすぎて“自力”が育たない | 不変条件「最後は本人」「自力で終える」を必須化。自力正解だけ習熟度を上げる。 |
| 単調になる | 同型を連続させすぎない／ほめの言い回しを多様化。 |
| 盛りすぎ（math-dialogue化） | 1画面1アクションを死守。ゲーム要素を入れない。 |
| エンジン共有の結合 | UI非依存の純ロジックだけを共有パッケージ化。データ形は現状を踏襲。 |

## 12. 用語集
- **助けのはしご**：つまづいた時の5段の手助け（ヒント/お手本/穴うめ/動画/類題）。
- **フェーディング**：お手本の足場を少しずつ外し、自力に近づける手法。
- **誤答パターン付き4択**：まちがいの選択肢に「つまづきの正体」を仕込み、選んだ瞬間に診断する仕組み。
- **つまづきマップ**：スキル別・タグ別の弱点を可視化したもの。
- **共有エンジン**：math-dialogue と「とけた！」が共通で使う、UI非依存の純ロジック（問題生成・スキル・診断・はいち）。
